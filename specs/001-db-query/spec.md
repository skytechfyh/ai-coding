# Feature Specification: 数据库查询工具

**Feature Branch**: `001-db-query`
**Created**: 2026-02-23
**Status**: Draft
**Input**: User description: "这是一个数据库查询工具，用户可以添加一个 db url，系统会连接到数据库，获取数据库的 metadata，然后将数据库中的 table 和 view的信息展示出来，然后用户可以自己输入 sql 查询，也可以通过自然语言来生成sql查询。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 连接数据库并查看元数据 (Priority: P1)

用户希望能够添加数据库连接信息（数据库 URL），系统能够连接到目标数据库，并自动获取该数据库中的所有表（tables）和视图（views）的元数据信息。

**Why this priority**: 这是整个工具的基础功能，用户必须先连接数据库并获取元数据，才能进行后续的查询操作。没有数据库连接就无法使用任何功能。

**Independent Test**: 可以通过添加一个有效的数据库连接字符串，验证系统能否成功连接并返回正确的表和视图列表。

**Acceptance Scenarios**:

1. **Given** 用户有一个有效的 PostgreSQL 数据库连接字符串，**When** 用户输入连接字符串并点击连接，**Then** 系统成功连接数据库并在界面上显示所有表和视图的名称列表
2. **Given** 用户输入了一个无效的数据库连接字符串，**When** 用户点击连接，**Then** 系统显示错误信息，提示连接失败的原因（如：无法连接、认证失败、数据库不存在）
3. **Given** 用户已经连接了一个数据库，**When** 用户点击某个表或视图，**Then** 系统显示该表/视图的列信息（列名、数据类型、是否主键、是否可空等）
4. **Given** 用户已经连接了一个数据库，**When** 用户在搜索框中输入表名关键词，**Then** 系统过滤显示匹配的表和视图
5. **Given** 用户尝试连接一个需要 SSL 的数据库但未启用 SSL 选项，**When** 用户点击连接，**Then** 系统提示需要启用 SSL 或显示连接错误

---

### User Story 2 - 执行 SQL 查询 (Priority: P1)

用户希望能够在连接数据库后，手动输入 SQL SELECT 语句并执行查询，系统将查询结果以表格形式展示。

**Why this priority**: 这是用户的核心需求之一，允许用户直接编写 SQL 来查询数据，是工具的主要价值所在。

**Independent Test**: 可以通过输入一个简单的 SELECT 语句（如 `SELECT * FROM users LIMIT 10`），验证系统能否返回正确的结果并显示为表格。

**Acceptance Scenarios**:

1. **Given** 用户已经连接到数据库，**When** 用户在 SQL 编辑器中输入 `SELECT * FROM users` 并执行，**Then** 系统返回查询结果并以表格形式展示
2. **Given** 用户输入的 SQL 包含语法错误，**When** 用户执行该 SQL，**Then** 系统显示语法错误的具体信息（包括错误行号和错误位置），提示用户修正
3. **Given** 用户输入的 SQL 不是 SELECT 语句（如 INSERT、UPDATE、DELETE、CREATE、DROP），**Then** 系统拒绝执行并明确提示只能执行 SELECT 查询
4. **Given** 用户输入的 SELECT 语句没有 LIMIT 子句，**Then** 系统自动添加 `LIMIT 1000` 再执行
5. **Given** 用户执行了一个返回大量结果的查询，**Then** 系统分页展示结果（每页默认 100 行），并显示总行数和当前页码
6. **Given** 用户的查询执行时间过长（超过 30 秒），**Then** 系统显示超时提示，询问用户是否继续等待或取消查询
7. **Given** 用户的查询执行过程中出现数据库错误（如表不存在、权限不足），**Then** 系统显示数据库错误信息，帮助用户定位问题

---

### User Story 3 - 自然语言生成 SQL (Priority: P1)

用户希望能够用自然语言描述查询需求，系统自动生成对应的 SQL 语句并执行。

**Why this priority**: 降低使用门槛，不会写 SQL 的用户也能完成数据库查询，提升用户体验。

**Independent Test**: 可以通过输入 "帮我查询最近10个用户"，验证系统能否生成正确的 SQL 并返回结果。

**Acceptance Scenarios**:

1. **Given** 用户已经连接到数据库且已获取元数据，**When** 用户输入 "查询所有订单"，**Then** 系统根据表结构生成 SQL、执行并返回结果
2. **Given** 用户输入的查询描述不够明确（如 "查询用户数据"），**Then** 系统生成 SQL 后先展示给用户确认，用户确认后再执行
3. **Given** 用户输入的查询无法根据现有表结构生成 SQL，**Then** 系统返回错误信息，说明无法理解查询需求，并列出可用的表供用户参考
4. **Given** 用户对系统生成的 SQL 不满意，**When** 用户点击修改按钮，**Then** 用户可以直接编辑生成的 SQL 后再执行
5. **Given** 用户的自然语言查询包含模糊的时间表达（如 "最近"、"上周"），**Then** 系统在生成 SQL 时需要用户确认具体的时间范围
6. **Given** LLM 服务响应时间过长（超过 10 秒），**Then** 系统显示加载状态，用户可以选择取消或继续等待

---

### User Story 4 - 保存和管理数据库连接 (Priority: P2)

用户希望能够保存多个数据库连接配置，方便下次快速连接，而不需要每次都输入连接字符串。

**Why this priority**: 提升便利性，用户通常需要频繁连接同一个数据库，保存连接配置可以节省时间。

**Independent Test**: 可以通过保存一个连接配置，然后重新打开应用，验证该连接配置是否存在且可用。

**Acceptance Scenarios**:

1. **Given** 用户成功连接到一个数据库，**When** 用户点击保存连接并命名，**Then** 下次打开应用时可以在列表中找到该连接配置
2. **Given** 用户有多个保存的数据库连接，**When** 用户选择其中一个并连接，**Then** 系统使用保存的配置快速连接
3. **Given** 用户想删除一个不再使用的连接配置，**When** 用户点击删除按钮并确认，**Then** 该连接配置被永久删除
4. **Given** 用户想修改一个已保存的连接配置，**When** 用户点击编辑按钮，**Then** 用户可以更新连接字符串或显示名称
5. **Given** 用户保存的连接在下次使用时发现连接信息已失效，**Then** 系统提示连接失败并询问是否更新连接信息

---

### User Story 5 - 查询历史和结果导出 (Priority: P3)

用户希望能够查看过去的查询历史，并将查询结果导出为常用格式以便进一步分析。

**Why this priority**: 方便用户追溯之前的查询操作，提高工作效率。

**Acceptance Scenarios**:

1. **Given** 用户执行过查询操作，**When** 用户打开查询历史面板，**Then** 系统显示最近 50 条查询记录（包含 SQL 语句、执行时间和结果行数）
2. **Given** 用户想重新执行历史记录中的某条查询，**When** 用户点击历史记录，**Then** SQL 自动填充到编辑器中
3. **Given** 用户查询得到了结果，**When** 用户点击导出按钮，**Then** 用户可以选择导出为 CSV、JSON 或 Excel 格式
4. **Given** 用户导出的数据量很大（超过 10000 行），**Then** 系统提示分批导出或压缩下载

---

### Edge Cases

- **数据库连接超时或网络不可达**: 显示具体超时错误，提示检查网络或数据库地址是否正确
- **用户权限不足**: 显示权限不足的具体信息，提示联系数据库管理员
- **数据库中存在大量表和视图**: 支持分页展示（每页 50 条），并提供搜索过滤功能
- **SQL 查询结果数据量很大**: 自动分页，每页 100 行，支持跳转到指定页
- **用户中断正在执行的查询**: 提供取消按钮，用户点击后立即终止查询
- **LLM 服务不可用**: 显示服务不可用提示，建议用户手动编写 SQL
- **数据库连接断开**: 自动检测连接状态，断开后提示用户重新连接
- **空查询结果**: 显示 "查询返回 0 条结果" 的友好提示
- **查询结果包含特殊字符**: 正确处理和显示特殊字符（如换行符、制表符、二进制数据）
- **数据库中有同名列**: 在结果表格中显示完整的列名（包括表别名前缀）
- **时间类型数据**: 以用户友好的格式显示日期时间
- **NULL 值处理**: 在表格中显示为空或特定标记（如 "NULL"），区分空字符串和 NULL

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持用户添加数据库连接字符串并建立连接
- **FR-002**: 系统 MUST 自动获取并展示连接数据库中的所有表和视图列表
- **FR-003**: 系统 MUST 展示用户选中表/视图的列信息（列名、数据类型、是否主键、是否可空）
- **FR-004**: 用户 MUST 能够手动输入 SQL SELECT 语句并执行
- **FR-005**: 系统 MUST 验证输入的 SQL 语法是否正确，使用 sqlparse 进行解析
- **FR-006**: 系统 MUST 拒绝执行非 SELECT 语句，并给出明确错误提示
- **FR-007**: 对于没有 LIMIT 子句的 SELECT 查询，系统 MUST 自动添加 `LIMIT 1000`
- **FR-008**: 系统 MUST 以 JSON 格式返回查询结果，前端负责渲染为表格
- **FR-009**: 用户 MUST 能够使用自然语言描述查询需求
- **FR-010**: 系统 MUST 将数据库连接信息和元数据存储到本地 SQLite 数据库中，以便复用
- **FR-011**: 当用户使用自然语言生成 SQL 时，系统 MUST 将表和视图的元数据作为上下文传递给 LLM
- **FR-012**: 系统 MUST 支持查询结果分页展示，每页默认 100 行
- **FR-013**: 系统 MUST 允许用户保存多个数据库连接配置
- **FR-014**: 系统 MUST 记录用户的查询历史（最近 50 条）
- **FR-015**: 系统 MUST 支持导出查询结果为 CSV、JSON 格式

### Non-Functional Requirements *(optional)*

- **性能**: 简单 SELECT 查询的响应时间应控制在 5 秒以内
- **可用性**: 关键操作（连接、查询、导出）都应有明确的加载状态和成功/失败反馈
- **错误处理**: 所有错误都应以用户可理解的方式呈现，避免显示原始的技术错误堆栈
- **数据安全**: 连接字符串在本地存储时应加密或采取适当保护措施

### Key Entities

- **数据库连接**: 连接ID、连接名称、连接字符串（需加密存储）、数据库类型、创建时间、最后使用时间
- **表/视图元数据**: 数据库连接ID、表/视图名称、类型（表/视图）、列信息列表（列名、数据类型、是否主键、是否可空、默认值）
- **查询记录**: 查询ID、关联的数据库连接ID、SQL 语句、执行时间、执行状态、返回行数、创建时间
- **查询结果**: 列定义（列名、数据类型）、行数据列表、总行数、查询耗时

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能够在 30 秒内完成数据库连接并获取元数据
- **SC-002**: 系统能够正确执行合法的 SELECT 查询并返回结果
- **SC-003**: 系统能够检测并拒绝非 SELECT 语句，错误提示准确率 100%
- **SC-004**: 对于没有 LIMIT 的查询，系统自动添加 LIMIT 1000 的成功率为 100%
- **SC-005**: 自然语言生成的 SQL 在简单查询场景下首次生成成功率达到 80%
- **SC-006**: 查询结果以表格形式展示，界面加载时间不超过 2 秒
- **SC-007**: 用户可以成功保存至少 10 个数据库连接配置
- **SC-008**: 查询历史能够正确记录并展示至少 50 条历史记录
- **SC-009**: 导出功能能够正确导出 CSV 和 JSON 格式的文件

## Assumptions

- 初始主要支持 PostgreSQL 数据库
- 使用 OpenAI API 作为 LLM 提供商
- 元数据存储在本地 SQLite 数据库文件中
- 不需要用户认证，所有功能对所有用户开放
- 用户具备基本的 SQL 知识（对于手动编写 SQL 的场景）
- 网络环境稳定，能够正常访问目标数据库和 LLM 服务
