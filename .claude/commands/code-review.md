You **MUST** consider the user input before proceeding (if not empty).

## Goal

对指定的 Python 或 TypeScript 代码进行深度审查，从以下维度进行评估并生成详细的审查报告：

1. **架构和设计** (Architecture & Design)
2. **KISS 原则** (Keep It Simple, Stupid)
3. **代码质量** (Code Quality: DRY, YAGNI, SOLID)
4. **性能** (Performance)
5. **Builder 模式** (Builder Pattern)

## Execution Steps

### 1. 解析输入确定审查范围

**输入格式**: 支持以下几种方式:

- **文件路径**: `path/to/file.py` 或 `path/to/file.ts`
- **目录路径**: `path/to/directory/` (递归审查所有 .py 和 .ts/.tsx 文件)
- **Glob 模式**: `src/**/*.py` 或 `src/**/*.ts`
- **多个路径**: `file1.py file2.ts src/`

**处理流程**:
1. 如果输入为空，提示用户输入审查目标
2. 解析输入，收集所有需要审查的文件
3. 排除以下文件:
   - `__pycache__`, `node_modules`, `.venv`, `venv`
   - 测试文件除非用户明确指定 (`test_*.py`, `*.test.ts`, `*.spec.ts`)
   - 生成的代码 (如 `gen_*.py`, `*.generated.ts`)
   - 第三方库代码
4. 对收集到的文件进行分类:
   - Python 文件 (.py)
   - TypeScript 文件 (.ts, .tsx)

### 2. 加载代码文件

对于每个需要审查的文件:
1. 读取文件完整内容
2. 使用 Read 工具逐个读取 (不要使用 glob 搜索后批量读取)
3. 记录文件路径、代码行数、函数/类数量

### 3. 执行深度审查

#### 3.1 架构和设计审查

**Python 特定检查**:
- [ ] 是否使用 dataclasses 或 Pydantic 进行数据建模？是否有不必要的 dict 或 raw types 使用？
- [ ] 是否使用 ABC (Abstract Base Class) 或 Protocol 定义接口？是否过度使用继承？
- [ ] 模块划分是否合理？是否有循环依赖？
- [ ] 是否使用 type hints？类型注解是否完整？
- [ ] 异常处理是否使用自定义异常类？是否有裸露的 except?
- [ ] 是否使用生成器进行惰性计算？是否适合场景？

**TypeScript 特定检查**:
- [ ] 是否使用 interface 定义对象结构？是否过度使用 type alias？
- [ ] 是否正确使用泛型？是否有 any 类型滥用？
- [ ] 是否使用 enum 或 const assertions？是否合理？
- [ ] 模块导出是否清晰？是否有 barrel 文件 (index.ts)？
- [ ] 是否使用 strict mode？tsconfig 配置是否严格？
- [ ] 是否有类型守卫 (type guards)？是否过度使用 unknown/any？

**通用架构检查**:
- [ ] 是否有清晰的单一职责 (SRP)？类/函数是否过长？
- [ ] 是否有依赖注入？依赖是否明确？
- [ ] 是否有适当的抽象层？抽象是否过度？
- [ ] 错误处理策略是否一致？
- [ ] 配置是否与代码分离？
- [ ] 是否有清晰的 API 接口设计？

#### 3.2 KISS 原则审查

**复杂度检查**:
- [ ] 函数是否过长？(Python > 50 行, TypeScript > 30 行为警告)
- [ ] 类是否过长？(Python > 300 行, TypeScript > 200 行为警告)
- [ ] 嵌套层级是否过深？(超过 3 层为警告)
- [ ] 条件判断是否过于复杂？(超过 3 个条件为警告)
- [ ] 参数是否过多？(超过 5 个参数为警告)
- [ ] 是否有不必要的注释或空行？
- [ ] 是否有过度抽象？(只有一个子类的抽象类)

**简化机会**:
- [ ] 是否有可以合并的相似函数？
- [ ] 是否有可以使用默认参数简化的地方？
- [ ] 是否有可以使用装饰器/高阶函数简化的地方？
- [ ] 布尔参数是否可以使用多态替代？

#### 3.3 代码质量审查

**DRY (Don't Repeat Yourself)**:
- [ ] 是否有重复的代码块？
- [ ] 是否有重复的逻辑可以抽取为共享函数？
- [ ] 是否有重复的配置？
- [ ] 是否有 magic strings/numbers 重复出现？

**YAGNI (You Aren't Gonna Need It)**:
- [ ] 是否有未使用的代码？(未调用的函数、未使用的参数)
- [ ] 是否有过度设计的功能？
- [ ] 是否有提前预留的"扩展点"？
- [ ] 是否有不必要的抽象层？

**SOLID 原则**:
- [ ] **S** - 单一职责: 每个类/函数是否只有一个变更理由？
- [ ] **O** - 开闭原则: 是否对扩展开放，对修改封闭？
- [ ] **L** - 里氏替换: 子类是否能够替换父类？
- [ ] **I** - 接口隔离: 是否有臃肿的接口？
- [ ] **D** - 依赖倒置: 是否依赖抽象而非具体？

**命名和风格**:
- [ ] 命名是否清晰表达意图？
- [ ] 命名是否一致？(snake_case vs camelCase)
- [ ] 是否有匈牙利命名法或无意义的命名？
- [ ] 代码风格是否一致？
- [ ] 是否有适当的空行和格式化？

**注释和文档**:
- [ ] 复杂逻辑是否有注释解释？
- [ ] 公共 API 是否有文档字符串？
- [ ] 注释是否与代码同步更新？
- [ ] 是否有TODO注释？是否需要处理？

**潜在错误**:
- [ ] 是否有空值/None/undefined 未处理的情况？
- [ ] 是否有竞态条件风险？
- [ ] 是否有资源泄漏风险？(文件、连接、内存)
- [ ] 是否有 SQL 注入风险？
- [ ] 是否有命令注入风险？
- [ ] 是否有 XSS 风险？
- [ ] 是否有硬编码的凭证？
- [ ] 是否有错误的异常处理？
- [ ] 是否有日志泄露敏感信息？

#### 3.4 性能审查

**算法复杂度**:
- [ ] 是否有 O(n²) 或更差的算法？(嵌套循环)
- [ ] 是否有不必要的全量加载？可以流式处理吗？
- [ ] 是否有重复计算？可以缓存吗？

**Python 特定**:
- [ ] 是否使用列表推导式替代 for 循环？
- [ ] 是否使用生成器处理大数据？
- [ ] 是否正确使用 `__slots__` 优化内存？
- [ ] 是否有不必要的全局变量？
- [ ] 字符串拼接是否高效？(使用 f-string 或 join)

**TypeScript 特定**:
- [ ] 是否使用 async/await 处理异步？
- [ ] 是否有不必要的同步阻塞？
- [ ] 是否正确使用 useMemo/useCallback 优化 React？
- [ ] 是否有不必要的对象/数组拷贝？
- [ ] 是否有不必要的 re-render？

**资源管理**:
- [ ] 是否有资源未正确释放？
- [ ] 是否有连接池未使用或配置不当？
- [ ] 是否有大量小对象创建？

#### 3.5 Builder 模式审查

**识别 Builder 模式**:
- [ ] 是否有复杂的对象构建逻辑？
- [ ] 是否有多个可选参数？
- [ ] 是否有方法链式调用？

**Builder 模式检查**:
- [ ] Builder 是否职责单一？
- [ ] 构建过程是否清晰？
- [ ] 是否有不必要的 Builder？
- [ ] 是否有 Builder 过度设计？
- [ ] 是否有可以简化为工厂函数的情况？
- [ ] Python 中是否使用 dataclass + __post_init__ 简化？

**复杂构建问题**:
- [ ] 构建步骤是否过多？(超过 10 步)
- [ ] 是否有循环依赖的构建？
- [ ] 是否有不必要的验证步骤？

- 提供快速修复的命令建议

## General Guidelines

### 审查原则

1. **实事求是**: 基于代码实际内容进行评估，不做假设
2. **建设性反馈**: 问题描述要具体，修复建议要可操作
3. **区分主次**: 区分必须修复和可选优化
4. **考虑上下文**: 考虑代码的实际使用场景

### 评分标准 (参考)

| 评分 | 含义 |
|------|------|
| 9-10 | 优秀，可作为范例 |
| 7-8 | 良好，只需小幅改进 |
| 5-6 | 一般，需要一些改进 |
| 3-4 | 较差，需要较大改进 |
| 1-2 | 很差，需要重写 |

### 注意事项

- 不要审查测试文件，除非用户明确要求
- 不要审查第三方库代码
- 对于代码生成工具生成的代码，标注并跳过详细审查
- 考虑 Python 和 TypeScript 的语言特性差异
- 对于每种语言，使用相应的最佳实践标准
